# 不動産DX統合SaaSプロジェクト (Go バックエンド)

> **📝 Note**
> 本プロジェクトは、ProFactでの**模擬実務プロジェクト**として開発されたアプリケーションです。

## 📖 プロジェクト概要

不動産仲介業者が抱える「資料や業務のIT導入」「アナログな商習慣」「情報の分散」「追客の属人化」というペイン（課題）を解消するための BtoBtoC型 SaaS です。
従来、紙ベースで行われていた物件図面（マイソク）の処理から、顧客コミュニケーション、案件のパイプライン管理、さらには電子契約までを、一つのプラットフォーム上でシームレスに完結させるシステムを目指しています。

本リポジトリでは、そのシステムのコアとなる**バックエンド API（Go言語）** を構築しています。

---

## 🛠 技術スタック

本プロジェクトは、保守性・拡張性を高めるため、以下の技術スタックとアーキテクチャを採用しています。

- **言語:** Go (1.23)
- **Webフレームワーク:** Gin
- **ORM (データベース操作):** GORM
- **データベース:** MySQL 8.0 (Docker環境)
- **アーキテクチャパターン:** クリーンアーキテクチャ (Clean Architecture)
- **API仕様:** OpenAPI 3.0によるスキーマ駆動開発 (Schema-Driven Development)

### クリーンアーキテクチャの採用理由

今後、外部システム（OpenAI APIによるOCR解析、電子契約APIなど）との複雑な連携が増えることを想定しています。そのため、コアとなるビジネスロジックをフレームワークやデータベースから切り離し（関心事の分離）、変更に強いシステム構成にしています。

## 📂 ディレクトリ構成（初学者向けガイド）

Goにおける標準的なプロジェクト構成とクリーンアーキテクチャを組み合わせています。

```text
.
├── cmd/
│   └── server/         # アプリケーションのエントリーポイント (main.go)
├── db/                 # データベースの初期化スクリプトやDDL (01_ddl.sql 等)
└── internal/           # 他のプロジェクトからインポートできないコア領域（Clean Architectureの中心）
    ├── domain/         # エンティティ(User, Deal等)とRepositoryのインターフェース。DBや外部を一切知らない。
    ├── usecase/        # アプリケーション固有のビジネスロジック（例: 「案件のステータスを更新する」等）
    ├── repository/     # domainのインターフェースの実装。GORMなどを用いて実際にDBとやり取りする層。
    └── handler/        # HTTPリクエストの受け取り・バリデーション、レスポンス返却のみを行う層（Ginなどを利用）。
```

> **💡 ポイント**: 処理の流れは基本的に `handler` ➡️ `usecase` ➡️ `repository` ➡️ `DB` と進み、逆向きの依存関係は作らないように設計しました。

## 🚀 実装しているもの・現状のステータス

本システムは導入リスクを最小化するため、段階的に実装を進める想定で設計されています。
**現在、土台となるコア機能（Phase 1）のAPI構築** を中心に行なっています。

### 主な機能

- **顧客管理機能 (Customers)**
  - 顧客の基本情報、希望条件（予算・エリア等）などの登録・検索・編集機能。
- **物件管理機能 (Properties)**
  - 物件の基本情報などの登録・検索・編集機能。
- **案件管理機能 (Deals) ※カンバン方式**
  - 「反響」〜「契約」までのステータスを持つ案件の進捗管理。顧客や物件、担当者を紐付けて管理します。

_(※ 以降のフェーズとして、AIを用いたマイソクOCRの自動データ化、電子契約API連携、自動日程調整機能などの実装が計画されています)_

## 🔌 API エンドポイント一覧 (Phase 1)

現在用意されている（または実装中の）主要な REST API エンドポイントは以下の通りです。JSONデータでやり取りします。

### 👥 顧客 (Customers)

| メソッド | エンドポイント        | 説明                               |
| :------- | :-------------------- | :--------------------------------- |
| `GET`    | `/api/customers`      | 顧客の一覧を取得します             |
| `POST`   | `/api/customers`      | 新しい顧客を登録します             |
| `GET`    | `/api/customers/{id}` | 指定した顧客の詳細情報を取得します |
| `PUT`    | `/api/customers/{id}` | 指定した顧客の情報を更新します     |
| `DELETE` | `/api/customers/{id}` | 指定した顧客を削除します           |

### 🏠 物件 (Properties)

| メソッド | エンドポイント         | 説明                               |
| :------- | :--------------------- | :--------------------------------- |
| `GET`    | `/api/properties`      | 物件の一覧を取得します             |
| `POST`   | `/api/properties`      | 新しい物件を手動登録します         |
| `GET`    | `/api/properties/{id}` | 指定した物件の詳細情報を取得します |
| `PUT`    | `/api/properties/{id}` | 指定した物件の情報を更新します     |
| `DELETE` | `/api/properties/{id}` | 指定した物件を削除します           |

### 📋 案件 (Deals)

カンバンボード上で管理する案件チケットの操作用APIです。

| メソッド | エンドポイント           | 説明                                                     |
| :------- | :----------------------- | :------------------------------------------------------- |
| `GET`    | `/api/deals`             | 案件一覧を取得します（カンバンボード描画用）             |
| `POST`   | `/api/deals`             | 案件を新規作成します（顧客・物件の紐付け）               |
| `PATCH`  | `/api/deals/{id}/status` | 案件の進行ステータス（ドラッグ＆ドロップ時）を更新します |
| `PUT`    | `/api/deals/{id}`        | 案件の詳細情報を更新します                               |
| `DELETE` | `/api/deals/{id}`        | 案件を削除します                                         |

## 🎯 何を可能にするのか（システムの提供価値）

本アプリケーションの導入により、不動産仲介業の現場に以下の変化をもたらします：

1. **業務効率化（脱アナログ化）**
   情報のデータ化と一元管理により、入力作業の負担を減らし、完全なペーパーレス化と業務効率の大幅な向上を実現させます。
2. **組織的な営業力の強化（ブラックボックス化の解消）**
   顧客情報の希望条件ステータスや行動ログをシステムに集約。担当者にしか分からない状態を防ぎ、チーム全体で効率的かつ組織的な追客が可能になります。
3. **チームマネジメントの強化**
   案件ごとの進捗状況をカンバンボードでリアルタイム可視化。マネージャーが適切にリソース配分やボトルネック特定を行えるようにします。
